# proto allowlist annotation で Tool 定義を自動生成する

エージェントから呼ばせたい既存 RPC を LLM の Function Calling Tool として公開するとき、proto に `(ai.tool.expose) = true` の annotation を付けて protoc で Tool schema を自動生成するパターン

## 動機

LLM に Tool を公開するとき、Tool 定義と既存 API 定義の二重管理が発生しがち

- proto には RPC として定義されている
- Tool definition は yaml / Go struct で別途書く
- 片方だけ更新されて drift する
- allowlist 漏れで本来公開すべきでない管理 API が LLM から叩かれる事故

## proto annotation で解決

```proto
syntax = "proto3";

package mycompany.v1;

import "ai/tool.proto";

service DataService {
  // 参照系：LLM に公開
  rpc GetCompany(GetCompanyRequest) returns (Company) {
    option (ai.tool.expose) = true;
    option (ai.tool.description) = "会社 ID から会社情報を取得する。社名・住所・業種を返す";
  }

  // 更新系：LLM に公開しない
  rpc UpdateCompany(UpdateCompanyRequest) returns (Company);
}
```

- `(ai.tool.expose) = true` が付いた RPC だけが Tool 化される
- `(ai.tool.description)` で LLM 向けの用途説明を書く（LLM が Tool 選択に使う）
- 付け忘れたら Tool 化されない（デフォルト deny）

## 自動生成パイプライン

protoc plugin（`protoc-gen-ai-tool` 相当を自作 or OSS）を挟む

```
*.proto
  → protoc --ai-tool_out=. --ai-tool_opt=...
  → tools_gen.go（Tool schema + binding code）
  → Genkit DefineTool で読み込み
```

生成物の例（Genkit Go）

```go
// Code generated by protoc-gen-ai-tool. DO NOT EDIT.

var GetCompanyTool = genkit.DefineTool(g, "get_company",
  "会社 ID から会社情報を取得する。社名・住所・業種を返す",
  func(ctx context.Context, req *datapb.GetCompanyRequest) (*datapb.Company, error) {
    return dataClient.GetCompany(ctx, req)
  },
)
```

- Tool 名は RPC 名から snake_case 変換（`GetCompany` → `get_company`）
- 入出力スキーマは proto message からそのまま JSON Schema 化
- bind 先は既存の ConnectRPC クライアントで統一（認証ヘッダも pass-through）

## allowlist 設計の勘所

### 1. デフォルト deny

`(ai.tool.expose) = true` が明示的に付いている RPC だけ公開。annotation 無し = 非公開

CI で `proto-check` を走らせて「annotation の付いた RPC がいくつか」を PR diff に出すと、意図しない追加が気付きやすい

### 2. read / write の分離

書き込み系 RPC は別 annotation（例 `(ai.tool.require_confirmation) = true`）を付けて、クライアント側で明示的な人間承認なしには呼ばせない設計にする

`UpdateCompany` のような破壊的操作が LLM 幻覚で呼ばれるリスクを構造的に防ぐ

### 3. audience ごとの allowlist

上の annotation に `(ai.tool.audience) = ADMIN` を追加して、audience 単位で公開範囲を制御できるようにしておくと、同じ RPC を admin には見せ user には見せない、といった設計が proto 1 箇所で完結する

### 4. 説明文の品質が Tool 選択精度を決める

LLM は `description` を見て Tool を選ぶ。雑な description（「Company を取得」など）だと誤選択が増える

- **入力の形式**（何を渡すか）
- **出力の内容**（何が返るか）
- **使い所**（どういう質問で使うか）

の 3 点を 1-2 文で書くのが目安

## MCP Tool 定義との対比

[Model Context Protocol](https://modelcontextprotocol.io/) の Tool 定義も同じ問題に直面する。MCP Server 側で Tool を列挙するとき、既存の REST/gRPC API を手で写経する運用になりがち

proto allowlist 方式と MCP は併用できる

- 社内 API（proto 定義あり）→ proto annotation で自動生成
- 外部 API（OpenAPI / REST のみ）→ MCP Server でラップ
- どちらも Genkit / Agent SDK から見れば同じ `Tool` インターフェース

どちらかを「正」として他方を自動生成する設計も可能（proto → MCP Server auto-gen など）

## 参考

- Genkit Go `DefineTool`: Tool の signature は `func(ctx, Input) (Output, error)` 固定
- protoc custom annotation: `extend google.protobuf.MethodOptions { bool expose = 50000; }` で独自フィールドを追加
- Function Calling 一般: OpenAI / Anthropic / Gemini すべて JSON Schema でスキーマを受ける
