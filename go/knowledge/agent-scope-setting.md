# エージェントのスコープ設定

エージェントの成否を分ける最初の判断。「何を任せるか」の境界線をどこに引くかで、評価可能性 / 価値貢献 / 実装複雑度のすべてが変わる。

## 一言で

スコープ設定は綱渡り。狭すぎても広すぎても曖昧でも失敗する。**「明確で境界のはっきりしたワークフロー」** に絞ることで、入力が具体 / 出力が構造化 / 短サイクル検証可能になる。

## 3 つの罠

### 罠 1: 狭すぎる

特定の処理だけに絞りすぎると、ボリュームの大きいリクエストを逃し、実務貢献が限られる。

**症状**: 「動くけど影響が小さい」「使われない」

### 罠 2: 広すぎる

「全部自動化する」「あらゆる問い合わせに対応」といった目標は、無数の例外に追われる。

**症状**: 「PoC は動いたけど production で破綻」「例外対応コードが本体より大きい」

### 罠 3: 曖昧すぎる

「顧客満足度を上げる」「業務効率を改善」といった目的は、成功の基準が見えない。

**症状**: 「効果があったかどうか分からない」「議論が平行線」

## 良いスコープの 3 条件

### 1. 入力が具体的

「何が来たら起動するか」が明確に書ける。

- ✅ 「特定 URL が渡されたら」
- ✅ 「特定コマンドが実行されたら」
- ❌ 「ユーザーから何か質問が来たら」

### 2. 出力が構造化

「何が出力されたら成功か」が定義できる。

- ✅ 「PR が作成され、URL が返る」
- ✅ 「SQL 文字列が出力され、人間がレビュー可能」
- ❌ 「いい感じの回答が返る」

### 3. 短サイクルで検証可能

入力 → 出力の写像を「golden test として N 件」用意でき、回帰検出ができる。

- ✅ 「過去の依頼 10 件で同じ出力が出るか」
- ✅ 「PR の diff が期待差分と一致するか」
- ❌ 「ユーザーアンケートで満足度が 3 ヶ月後に上がるか」

## スコープを絞り込む手順

1. **業務フローを書き出す**: 現状人間が手作業でやっている定型処理を列挙
2. **頻度 × 工数で並べる**: 上位の「頻度高 × 1 件あたり工数大」がスコープ候補
3. **入力 / 出力の明確さで絞る**: 構造化できるものを優先
4. **検証可能性で絞る**: golden test を作れるか
5. **MVP として 1 つに絞る**: 最初は 1 ユースケースで完成度を上げる

## アンチパターン

| パターン | 症状 | 対処 |
|---|---|---|
| 「とりあえず全社の業務を AI 化」 | 例外地獄 + 評価不能 | ユースケース別に分解、1 つずつ |
| 「ユーザー体験向上」 | 評価指標がない | 具体的な KPI に落とす |
| 「○○ チームの作業を自動化」 | チーム内のあらゆる作業が対象 → 過剰スコープ | 1 タスクに絞る |
| 「すごい AI を作る」 | 何も具体化されない | 入力 / 出力 / 検証の 3 条件で具体化 |

## ROI とスコープの関係

| スコープ × 頻度 | ROI | 判断 |
|---|---|---|
| 狭い × 高頻度 | 高 | **最優先で狙う** |
| 狭い × 低頻度 | 低 | やる意味薄い |
| 広い × 高頻度 | 破綻リスク高 | 分解してから |
| 広い × 低頻度 | 最悪 | 着手しない |

最初は **狭い × 高頻度** だけ狙う。広げるのは後。

## 5 原則との関係

`agent-system-design-principles.md` の 5 原則と直結:

- **継続的学習**: 検証可能なスコープでないと改善ループが回らない
- **モジュール性**: 1 ユースケース = 1 モジュール、として設計すれば追加 / 入れ替えが容易
- **拡張性**: 狭いスコープで成功 → 似たユースケースに横展開、というスケール戦略

## 良いスコープの実例

| エージェント | 入力 | 出力 | 検証 |
|---|---|---|---|
| repo-reader | リポジトリパス | アーキテクチャ要約 Markdown | 既知リポで再現 |
| PR draft 作成 | 依頼テキスト + 既存ファイル | diff + PR description | 過去 PR で再現 |
| 障害調査 | エラーメッセージ + ログ範囲 | 原因仮説 + 対処手順 | 過去インシデントで再現 |

すべて「狭く、具体的で、構造化された出力、検証可能」。これが production 投入できるエージェントの最小単位。
